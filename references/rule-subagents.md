# 独立通道与职责分配规则

## 固定用户可见名称

- `生成内容`：页纲、样片和完整 PPTX/HTML 的生成或重大修改。
- `检查风格`：视觉、模板、版式、安全区、渲染、对比度、Logo、页脚、背景和页面节奏。
- `检查文字`：中文表达、错别字、逻辑、来源忠实度、标题准确性、密度和拆页建议。

## 默认策略

单 Agent 是默认执行方式。不要为每个已加载 rule 创建独立子智能体；规则是检查清单，通道是端到端责任单元。

满足任一条件时才启用独立通道：

- 正式对外交付或用户明确要求独立复核；
- 覆盖、批量修改、重要品牌材料等高风险任务；
- `生成内容`、`检查风格`、`检查文字` 三条通道均适用、所有权不重叠且合并标准清晰。

## 分配矩阵

| 规则族 | 主要通道 | 可合并条件 |
| --- | --- | --- |
| intake、conversion、page structure | `生成内容` | 小任务可由主 Agent 执行 |
| template、layout safety、page styles、render validation | `检查风格` | `check_only` 可直接作为主检查通道 |
| source fidelity、copy、density | `检查文字` | 没有文案变更时可不启用 |
| guardrails、failure recovery、SSOT | 主 Agent/合并者 | 不替代风格或文字检查 |

## 职责分离与真实性

- 正式或高风险任务中，生成者不能批准自己的最终输出。
- 独立通道必须拥有不同执行上下文；同一 Agent 的先后两遍只能称“分离自检”。
- 工具不支持独立子智能体时，记录限制、执行分离自检并保留检查证据。
- 不适用的通道写 `not_applicable` 和理由，不虚构人员或运行记录。
- `check_only` 不需要生成通道；`handoff` 不启用三通道。

## 通道交接

每个适用通道输出：范围、输入、执行检查、PASS/WARN/FAIL、阻塞项、建议、证据路径。冲突提交 SSOT 合并者，不能互相覆盖结论。
