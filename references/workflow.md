# V1.2 主工作流

`SKILL.md` 负责入口路由，本文件负责模式化执行。先读 `rule-routing-and-modes.md`，再按模式只加载必要规则。

## 阶段 0：启动、模式与边界

1. 说启动语。
2. 在扫描工作区之前执行明确意图门；纯激活语返回 `E_ROUTE_AMBIGUOUS` 并停止，不读取目录材料。
3. 选定 `check_only`、`repair`、`convert`、`create` 或 `handoff`，记录目标和禁止事项。
4. 按 `rule-intake.md` 确认本次输入及其归属；未确认的工作区文件不得读取正文或参与路由。
5. 只识别已确认输入的文件类型；不重复询问其中可直接给出的页数、标题、图片和比例。
6. `handoff` 直接形成移交记录并停止演示稿生成。

## `check_only` 路径

1. 保持源文件只读，定义检查范围和证据形式。
2. 按需读取模板、结构、排版与校验规则，不创建无关 `Design.md` / `Content.md`、生成日志或样片。
3. 运行适用检查器并渲染/预览；无法渲染时标记降级。
4. 输出逐页或逐问题的 PASS/WARN/FAIL、证据、限制和建议，不执行修复。

## `repair` / `convert` / `create` 路径

### 1. 最小澄清与不可写门

1. 先验证本次输入归属；候选材料未确认时使用 `E_INPUT_UNCONFIRMED`，不得创建生成日志。
2. `create`、`convert`、重大 `repair` 必须按 `rule-intake.md` 补足最小字段，并按 `rule-clarification-and-design-gates.md` 展示解析 Brief。即使输入完整也必须等待用户确认 Brief。
3. 展示 `rule-template-selection.md` 中的模板与 9 套视觉风格预览卡；用户可选择具体项或“智能推荐”，但推荐结果仍须确认。
4. Brief、模板与风格未分别为 `confirmed` 时，不得创建完整 PPTX/HTML、写入完整演示稿、静默选择默认模板/风格或开始完整生成。仅可展示澄清问题、Brief、选择卡、页纲或样片提案。
5. PPT/PDF 转换确认“修改内容并套模板”或“只套模板不改内容”。Logo 替换、源文件覆盖和图片资产来源策略均需明确决定。

### 2. 内容与渐进加载

1. 读取任务所需的已确认来源。
2. 先读设计索引与共享规则，再只读已确认的场景、模板和视觉风格文件。
3. 按 `rule-content-expression.md` 给出页数、逐页目的、表达类型、视觉载体、资产来源、风格约束、文本预算和拆页策略；核心内容页优先采用有信息作用的图片、图解、图表或截图，而非装饰性配图。

### 3. 日志、样片与确认

1. `create`、`convert` 和重大 `repair` 在写样片或完整输出前创建 V1.2 生成日志，并记录确认态和资产来源策略。
2. 生成一页代表性或最高风险样片并渲染检查；样片应体现已确认的模板、风格与页级表达策略。
3. 等待用户确认样片后再完整生成。只有用户在 Brief 与模板/风格均已确认后，明确说明“跳过样片”并给出理由时，才可记录 `explicitly_skipped` 继续；不可用“赶时间”“默认”“智能推荐”推定跳过。

### 4. 完整生成或修复

1. 默认不覆盖来源，先写版本化新文件。
2. 新建或转换的 PPTX 应遵守固定页与结构规则；固定页是否适用仍取决于场景规则。
3. 文本先做预算，再进版；特殊页型按 `rule-page-styles.md`，视觉表达按 `rule-content-expression.md`。
4. 正式或高风险任务按 `rule-subagents.md` 分配独立通道；其余任务使用单 Agent 的分离步骤。

### 5. 验证与返工

1. PPTX 运行 `check_pptx_layout.py`；HTML 运行 `check_html_layout.py`。
2. 渲染/预览每页，处理全部未批准 `FAIL`；对 WARN 给出判断。
3. 失败时按 `rule-failure-recovery.md` 分类，只修最小责任层并重跑针对性检查。
4. 最后执行完整交付门禁；正式或高风险任务合并 SSOT。

## 交付

最终回复给出：模式、产物或被检查文件、确认态、所选模板/风格、资产来源策略、预览证据、日志/SSOT（若适用）、检查命令与结论、降级或遗留问题。不得让结构门禁、模型行为与结果质量三种证据互相替代。
