# V1.2 输入归属与最小澄清规则

本文件只定义输入的本次归属与最小信息收集。`create`、`convert`、重大 `repair` 的 Brief、模板/风格和样片确认态由 `rule-clarification-and-design-gates.md` 定义；不得以本文件的“少问原则”跳过那些门。

## 本次输入归属门

模式明确后、读取材料内容前，先建立“本次已确认输入”。以下任一条件成立才算已确认：

1. 用户在当前请求中粘贴内容或附加文件；
2. 用户在当前请求中明确点名文件、路径或“本目录的 Design.md / Content.md”；
3. Agent 只展示候选文件名和修改时间后，用户明确选择了其中的材料；
4. 用户明确要求继续当前同一任务，且相关输入已在当前任务中确认过。长期记忆、旧任务摘要或另一任务的确认不算本次确认。

目录中恰好存在的 `Design.md`、`Content.md`、来源文件、历史日志和既有产物都只是候选。`scripts/scan_task_context.py` 只做发现，不能授予使用权限；文件存在、名称匹配、修改时间较新或记忆中曾被使用，都不能单独证明它是本次输入。

- 未给出明确执行意图：先返回 `E_ROUTE_AMBIGUOUS`，不得扫描或读取工作区内容。
- 已明确 `create` / `convert` / `repair`，但没有任何已确认输入：返回 `E_INPUT_MISSING`；若工作区存在候选材料但用户未选择，返回 `E_INPUT_UNCONFIRMED`。只问一个最小问题，不读取候选材料正文、不创建日志、不生成产物。
- 用户确认后，在生成日志及适用的 SSOT 中记录输入路径和归属类型：`current_turn`、`explicit_path`、`confirmed_workspace` 或 `confirmed_continuation`。
- 当前请求本身已包含主题、提纲或正文时，它属于 `current_turn`，无需为了形式再次确认。

## 最小澄清字段

对 `create`、`convert`、重大 `repair`，收集以下字段中尚不能从**已确认输入**或当前请求确定的项目。可在一轮合并提问中询问所有缺项；已有明确答案不得重复问。

| 字段 | 最小可接受答案 | 何时必问 |
| --- | --- | --- |
| 场景与受众 | 使用场景，以及主要受众 | 两者任一未知 |
| 目标与核心主张 | 希望受众理解、相信或决定什么 | 主题或原始材料无法确定 |
| 输出规格 | PPTX/HTML、比例和预期页数或页数范围 | 任一未知 |
| 内容处理边界 | 保真、可重排、可改写、可删减/补充的边界 | 转换、修复或素材密集任务未知 |
| 模板与视觉风格意图 | 具体选择，或明确要求“推荐后由我确认” | 始终在设计选择门中显式呈现 |
| 图片资产来源策略 | 仅用户素材、可用确认的本地资产、可用图解/图表、是否允许 AI 生成；外部检索一律不自动启用 | 内容页需要视觉资产且策略未知 |
| PPT/PDF 处理模式 | `修改内容并套模板` 或 `只套模板不改内容` | PPT/PDF 转换或套模板 |

图片资产来源策略不等同于图片风格：前者决定允许使用哪些来源，后者由用户确认的视觉风格决定。不得因“多用图片”自动搜索、下载或接入外部素材服务；没有适合图片资产时，优先使用遵循所选风格的图解、图表或结构化视觉表达。

## 数字菜单

所有菜单接受纯数字确认；数字只映射到最近一次尚未过期的菜单：

- 场景：`1. GTLC 大会` / `2. TGO日常活动分享`
- PPT/PDF 处理：`1. 修改内容并套模板` / `2. 只套模板不改内容`
- 输出格式：`1. PPT` / `2. HTML` / `3. 两者`

模板和 9 套风格的数字菜单、预览卡与“智能推荐”确认规则见 `rule-template-selection.md`。用户只输入 `1` 时，必须引用最近一次菜单的第一个选项；不允许把数字解释为未展示的默认风格。

## 少问原则

只问会改变结果、不能从已确认材料直接推出的最少问题。少问不等于不确认：即使字段齐全，仍须展示解析 Brief，并等待 Brief、模板与风格的明确确认后才可写入完整演示稿。
