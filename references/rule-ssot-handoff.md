# SSOT 合并与交接规则

SSOT 是本次任务唯一事实源。不同规则子智能体只能提交独立结论，不能互相覆盖；最终由 SSOT 合并者统一裁决、合并和交接。

## SSOT 文件

每次生成、改造或复杂修复任务，在生成日志旁维护 SSOT 区块。可以直接写入生成日志，也可以在同目录创建：

```text
gen-tgo-ppt-SSOT-YYYYMMDD-HHMMSS.md
```

如果创建独立 SSOT 文件，生成日志必须记录该路径。

## 子智能体交接格式

每个规则子智能体必须输出同一结构：

```markdown
## 规则交接：<规则文件名>

- 子智能体：
- 规则文件：
- 输入材料：
- 适用范围：
- 已执行检查：
- 通过项：
- 风险项：
- 阻塞项：
- 建议修改：
- 需要其他规则确认：
- 产物或证据路径：
- 结论：PASS / WARN / FAIL
```

## 合并原则

- SSOT 只保留一个最终任务状态，不保留互相冲突的多份真相。
- `FAIL` 优先级高于 `WARN`，`WARN` 高于 `PASS`。
- 规则之间冲突时，以更保守、更能防止交付事故的规则为准。
- 固定页、排版安全、Logo/页脚禁入区、文本溢出、用户明确要求优先级最高。
- 如果两个规则给出不同建议，SSOT 合并者必须记录冲突、裁决理由和采纳方案。
- 未被 SSOT 采纳的建议不能直接进入最终产物。

## SSOT 必填区块

```markdown
# SSOT

## 任务目标

## 用户明确要求

## 输入材料

## 已加载规则

## 规则子智能体分工

## 规则交接摘要

## 冲突与裁决

## 当前唯一执行计划

## 产物路径

## 校验结果

## 遗留问题
```

## 交付门槛

- 所有已加载规则必须有对应交接摘要。
- 所有 `FAIL` 必须修复或由用户明确批准例外。
- 最终回复只能引用 SSOT 中的最新产物路径、检查结果和遗留问题。
- 如果工具策略不允许创建多个子智能体，必须在 SSOT 和生成日志中记录“未满足独立子智能体执行”，并用分离自检替代，不能声称已完成独立执行。
