# SSOT 合并与交接规则

SSOT 用于正式或高风险的 `create`、`convert`、重大 `repair`，以及正式审计型 `check_only`。普通检查、轻微修复和 `handoff` 不强制创建完整 SSOT。

## 文件与范围

SSOT 可写在生成日志中，或创建 `gen-tgo-ppt-SSOT-YYYYMMDD-HHMMSS.md` 并由日志引用。它只记录当前任务事实，不承担长期 Skill 版本事实；Skill 版本与依赖以 `references/skill-manifest.json` 为准。

## 通道交接格式

```markdown
## 通道交接：<生成内容 / 检查风格 / 检查文字 / 主检查者>

- 执行上下文：
- 输入与范围：
- 已执行检查：
- 通过项：
- 风险项：
- 阻塞项：
- 建议修改：
- 证据路径：
- 结论：PASS / WARN / FAIL / not_applicable
```

## 合并原则

1. 用户明确要求优先，其次为安全/不可逆约束、模式合同、模板默认值。
2. 未批准 `FAIL` 阻塞交付；WARN 必须有人工判断。
3. 冲突必须记录双方结论、裁决理由和唯一采纳方案。
4. 不能把同一 Agent 的分离自检写成独立子智能体验证。
5. 最终回复只引用 SSOT 最新产物、检查结论和遗留问题。

## 必填区块

任务目标、模式、用户明确要求、输入材料、已加载规则、通道分工、交接摘要、冲突与裁决、唯一执行计划、产物路径、验证结果、证据层级、遗留问题。
