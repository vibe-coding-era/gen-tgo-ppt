# V1.1 路由与模式规则

本文件在 Skill 加载后首先读取。一次任务只能有一个主模式；需要变更模式时，记录原因后重新执行对应门禁。

## 两层路由合同

- 宿主触发层：由 `SKILL.md` frontmatter 的 `description` 决定是否加载本 Skill。正触发包括 TGO/GTLC PPTX 或 HTML 的生成、转换、修复与校验；普通写作、Excel/Word、通用设计、仅阅读 PDF 等任务应为 `no_trigger`。
- Skill 内执行层：Skill 已加载后，才在下列五种模式中选择。`handoff` 不能反向冒充宿主触发规则。

## 明确执行意图门

1. 在扫描或读取工作区材料之前，先确认用户是否给出了明确动作和预期产物。
2. “运行此 SKILL”、“使用 `$gen-tgo-ppt-skill`”或只点名/加载 Skill 属于激活语，不是 `create`、`convert`、`repair` 或 `check_only` 请求。
3. 激活语没有明确动作或产物时，返回 `E_ROUTE_AMBIGUOUS`，只问：“你希望新建、转换、修复，还是只检查 TGO/GTLC 演示稿？”然后停止本轮执行。
4. 处于 `E_ROUTE_AMBIGUOUS` 时，不得根据当前目录名称、目录中的文件、历史日志、既有产物、长期记忆或旧任务摘要推断模式；不得读取这些文件的内容、创建日志或生成演示稿。
5. 只有用户明确表达动作和产物后，才进入下方模式判定。未通过本门禁时不强行选择五种模式之一。

## 模式判定顺序

1. 用户只要求检查、审计、评价且明确不修改：`check_only`。
2. 用户要求修复或优化既有 PPTX/HTML：`repair`。
3. 用户要求把 PPT/PDF/Markdown/HTML/文本转换成另一演示格式：`convert`。
4. 用户要求从主题、提纲或素材新建演示稿：`create`。
5. Skill 已加载但任务不属于 TGO/GTLC 演示稿、缺少必要宿主能力或必须由其他专门能力处理：`handoff`。

若两个模式都可能成立，以用户明确动词和预期产物为准；仍有歧义时，只问一个会改变模式的问题并使用 `E_ROUTE_AMBIGUOUS`。工作区状态只能帮助执行已确认的模式，不能替代用户的执行意图。

## 单 Agent 与独立通道

- 默认由单 Agent 完成路由、执行和分离自检，避免为每个 rule 创建一个子智能体。
- 满足任一条件时启用独立通道：正式对外交付；覆盖/批量修改等高风险操作；生成、风格、文字三条可分离通道均实际适用且有明确合并标准。
- 独立通道名称为 `生成内容`、`检查风格`、`检查文字`。未使用独立上下文时必须写“分离自检”，不能写“独立验证”。
- `check_only` 不要求虚构生成通道；只启用与检查范围相符的通道。

## Handoff 输出

`handoff` 必须说明：越界原因、已确认输入、未执行操作、推荐接手能力/Skill/工具、下一步。不得创建空白 PPTX、伪造检查结果或声称已完成移交后的工作。
